## 開発の目的・解決した課題

### 課題
- AIに頼んでも「うーん」となる回答ばかり返ってくる
- プロンプトに「構造」がないと、AIは的外れな回答を返す
- でも毎回構造を意識して書くのは大変
- エンジニアでも非エンジニアでも、この課題は共通

### 解決
- プロンプトを **Role / Context / Task / Format** の4要素に分解する体験
- 既存プロンプトの「足りない要素」を可視化（診断モード）
- ゼロから組み立てる支援（ビルドモード）
- 5種類のテンプレート（コード生成・文章校正・アイデア発散・要約・翻訳）で誰でも入れる

## 詰まったこと・失敗したこと（コンテンツに使える素材）

### 1. 診断ロジックの判定基準を決めるのが一番難しかった
最初は「文字数」だけで判定しようとしたが、「あなたは○○です」だけ書いて他は空でもスコアが上がってしまうバグが発生。
最終的に「キーワード検出 + 文字数 + 組み合わせ」の3軸で評価する設計に落ち着いた。

### 2. XSS対策の罠
プレビューに色付け表示するために innerHTML を使いたかったが、ユーザー入力をそのまま入れると XSS の脆弱性になる。
escapeHtml 関数を作って、ユーザー入力は必ずエスケープを経由するようにした。

### 3. プロンプトの「悪い例」が思いつかない問題
良いプロンプトの例は無限に作れるが、「いかにもありそうな悪い例」を作るのが意外と難しかった。
結局「コード書いて」という3単語が一番リアルな悪い例だった。

### 4. 「Format」の判定キーワードが文化依存
「箇条書きで」「○文字以内」など、日本語の「形式指定」のバリエーションが思ったより多い。
キーワードリストの調整で時間を使った。

### 5. リリース前テストで「90点止まり問題」が発覚（広報チーム必読）
自分で書いたCARTE形式の説得メール用プロンプト（実務レベルの高品質プロンプト）を投げたら90点しか出なかった。
原因は職種辞書に「リーダー」「提案者」「部長」などビジネス系の職種が入っておらず、Roleが15点になっていたこと。
辞書を拡張してから100点が出るようになった。

**この体験そのものが「ツールは自分で育てる前提」というコンセプトを強化する素材になる。**
「ツール作って自分で投げたら90点だった。辞書を増やしたら100点になった。Forkして自分の業界の語彙を入れると、より精度が上がるはず」という投稿はリアルで強い。

## スクリーンショット見せ場シナリオ

### シーン1（最重要・必ず撮る）：診断モードのビフォーアフター
**操作：**
1. 診断モードを開く
2. 「悪い例を試す」ボタンを押す
3. → 「コード書いて」と表示され、スコア 15/100 が出る
4. 「構造化する →」を押す
5. → ビルドモードに自動遷移、4ブロックが色付きで埋まる

**スクショ：** 左に「コード書いて」とスコア15、右に構造化された4色ブロックのプロンプト

### シーン2：診断結果の品質スコアと「足りない要素」
**操作：**
1. 「良い例を試す」ボタンを押す
2. → スコア92点、全ブロック「○あり」が表示される

**スクショ：** 大きな「92 / 100」のスコアと4色のブロック判定

### シーン3：ビルドモードのテンプレート選択
**操作：**
1. ビルドモードのタブを開く
2. テンプレートチップ（コード生成、文章校正、アイデア発散、要約、翻訳）が並んでいる
3. 「⚡コード生成」を選ぶ

**スクショ：** 5種のテンプレートチップと、選択後の4ブロックが埋まっている状態

## デモデータ（投稿で使える）

### 悪い例
```
コード書いて
```
→ スコア15点

### 良い例
```
あなたはJavaScriptに精通したエンジニアです。
現在Next.js 14でブログサイトを作っています。
記事一覧をページネーションする関数を、TypeScriptで書いてください。
コードブロックで出力し、主要な処理にコメントをつけてください。
```
→ スコア92点

### コード生成テンプレートの完成プロンプト例
```xml
<role>
あなたは経験豊富なソフトウェアエンジニアです。コードの可読性と保守性を重視します。
</role>

<context>
使用言語：TypeScript
環境：Next.js 14 / App Router
制約：外部ライブラリは Next.js 標準のもののみ
</context>

<task>
URLを受け取ってOGP情報（title, description, image）を取得する関数を実装してください。
</task>

<format>
コードブロックで出力し、主要な処理に短いコメントをつけてください。エラーハンドリングも含めてください。
</format>
```

## 広報チームへのメモ（バズに使える視点・伊藤 拡斗向け）

### 投稿軸の推奨

**軸：「プロンプトの構造化を体験させるツール」**

このツールの本当のポイントは「Role / Context / Task / Format の4要素に分解する」という考え方そのもの。
えぬが組織編成（役割を持ったAIチーム）で動かしているのと同じ構造の話なので、
**Phase 2 種明かしへの伏線として完璧に機能する。**

### おすすめフック候補（広報チームへ）

1. 「AIに『コード書いて』って雑に頼んだ結果、構造化したら別人になった」（ビフォーアフター系）
2. 「プロンプトに足りないものを色で見せるツール、作った」（可視化の驚き）
3. 「AIへの指示、4つのパーツに分けるだけで結果が変わる」（教育系）
4. 「ChatGPTに思った答えが返ってこない人向け。プロンプトを診断するツール作った」（共感系）
5. 「自分が毎回やってるプロンプト設計の手順を、誰でも使えるツールにした」（実用系）
6. **「プロンプト診断ツール作って、自分の渾身のプロンプトを投げたら90点だった。辞書を増やしたら100点になった」**（実体験系・最強）

### 拡散層拡大の動線

- **エンジニア層**：「バニラJS / コード公開 / Fork可」の文脈で技術投稿に
- **非エンジニア層**：「ChatGPTを使ってる人へ」の文脈で実用ツール紹介に
- 投稿の本文構成を「最初に共感（AIに思った答え返ってこない）→ 解決（こういうツール作った）→ 動線（GitHub URL）」にすると両層に届く

### 投稿形式の推奨
- **単発 + 動画（GIF）**：ビフォーアフターを5秒で見せる
- スレッドは不要（一目で理解できる構造のため）
