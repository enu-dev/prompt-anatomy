## 1. プロジェクト背景

えぬから「最近のトレンドに乗った6作目を作りたい」という依頼を受け、広報チームのリサーチを経て選定された案。

選定経緯は `output/strategy/6th-project-research.md` 参照。

**えぬの追加意図（重要・Phase戦略に直結）：**

> 「AI組織のような編成があれば、こういう便利なツールを作れる」という認知を広めたい。Phase 2の種明かしへの伏線として機能させる。

**この意図を受けた制作方針：**
- ツール自体が「AIをどう使うか」の哲学を体現する設計にする
- プロンプトを「Role / Context / Task / Format」の役割で分解する構造は、えぬの組織編成（役割を持ったAIチーム）と同じ構造になる
- 高品質に仕上げる（モデルをOpusに切り替えての制作）

---

## 2. 何を作るか（プロダクト定義）

### プロダクト名（仮）
**プロンプト構造化ツール（prompt-anatomy）**

最終的なリポジトリ名候補：
- `prompt-anatomy`（推奨）
- `prompt-blocks`
- `prompt-structure`

### コア体験

ユーザーが**「AIから思った答えが返ってこない」**という課題に対し、**「プロンプトに構造を与えれば結果が変わる」**ことを体験できるWebツール。

```
雑なプロンプト    →   構造化されたプロンプト
「コード書いて」  →   <role>あなたは...</role>
                     <context>...</context>
                     <task>...</task>
                     <format>...</format>
```

### 2つのモード

**モード1：診断モード（Diagnose）**
- ユーザーが自分の「雑なプロンプト」を貼り付ける
- ツールが「足りない要素」を色付きで指摘（Role がない・Context が薄い・Format 指定がない 等）
- 改善ボタンで構造化された形に変換
- そのままClaude/ChatGPTにコピペできる

**モード2：ビルドモード（Build）**
- ゼロから組み立てる
- テンプレート選択（コード生成 / 文章校正 / アイデア発散 / 要約 / 翻訳）
- 各ブロックを埋めるだけで構造化プロンプトが完成
- リアルタイムでプレビュー表示

## 3. 必須機能

### 3.1 構造ブロック（4種）

| ブロック | 色（仮） | 役割 |
|---------|---------|------|
| **Role**（役割） | 紫 | AIに与える専門家/立場の設定 |
| **Context**（文脈） | 青 | 背景情報・前提知識 |
| **Task**（指示） | 緑 | 具体的にやってほしいこと |
| **Format**（形式） | 橙 | 出力形式の指定 |

### 3.2 診断モードの動作

1. テキストエリアにプロンプトを貼り付け
2. 「診断する」ボタンクリック
3. 4ブロックの「ある/ない」「強い/弱い」を判定して色付きで表示
4. 「品質スコア」を100点満点で表示（インパクト用）
5. 「構造化する」ボタンで自動変換

### 3.3 ビルドモードの動作

1. テンプレート選択（5種）
2. 各ブロックのテキストエリアに記入
3. 右側にリアルタイムプレビュー（完成プロンプト）
4. 「コピー」ボタンで全文コピー
5. XML形式 / プレーンテキスト形式 を切り替え

### 3.4 テンプレート（5種）

1. **コード生成**（エンジニア向け）
2. **文章校正**（ライター・ビジネス向け）
3. **アイデア発散**（企画職向け）
4. **要約**（学生・ビジネス向け）
5. **翻訳**（一般向け）

各テンプレートには「サンプル入力」と「ヒント」が初期表示される。

### 3.5 共通機能

- ワンクリックコピー（クリップボードAPI）
- レスポンシブ（PC・タブレット・スマホ）
- ダークモード基調（コードが映える）
- 完全オフライン動作（CDN1個まで許容、なしが理想）

## 4. あったら嬉しい機能（任意）

- 完成プロンプトのシェア機能（URLパラメータ）
- ブロックの並び替え（ドラッグ&ドロップ → 重いのでクリック移動でも可）
- 「良い例 / 悪い例」のサンプル比較表示
- localStorage で直近の入力を保存

---

## 5. スコープ外（やらない）

- 実際にAI APIに問い合わせる機能（バニラJSのみ制約）
- ユーザー登録・ログイン
- 多言語対応（日本語のみ）
- 履歴管理・複数プロンプト保存

---

## 6. 技術スタック制約

- HTML / バニラCSS / バニラJS（ES6+）のみ
- CDNライブラリは原則使わない（必須ならCTO承認）
- ファイル分離必須（`<style>` / `<script>` 埋め込み禁止）
- GitHub Pages公開前提（サーバーサイド処理なし）

---

## 7. 投稿時に強調するポイント（広報チーム向け）

優先度順：

1. **「AIから思った答えが返ってこない人へ」** ← 共感トリガー（非エンジニアにも刺さる）
2. **「プロンプトを構造化したら別物になる」** ← 体験できる驚き
3. **「エンジニアじゃなくても使える」** ← 拡散層拡大
4. **「全部バニラJSで作った」** ← エンジニアコミュニティ向け
5. **「Forkして自分のテンプレを増やせる」** ← 参加感の演出

## 8. スクリーンショット見せ場（必須）

**シーン2：**診断結果（品質スコアと足りない要素の指摘）

**シーン3：**ビルドモードのテンプレート選択画面

## 9. 納期感

- **品質優先**：今回はモデルをOpusに切り替えての制作のため、いつもよりクオリティを上げる
- 1〜2セッションでの完成を目標とするが、品質を犠牲にしない
- えぬが「これは出せる」と思える水準を達成すること
